# 通用创作工作流编排

## 核心原则

1. **subagent 无法与用户交互** — 它们是 fire-and-forget 的，启动后独立执行直到完成
2. **需要用户确认的环节必须由主agent直接完成** — 不能委托给 subagent
3. **自动化 chain 只包含无需交互的纯执行步骤**
4. **所有评审/质检结果必须展示给用户决策** — 不能静默跳过
5. **分阶段生产，分阶段确认** — 每个阶段独立产出、质检、确认
6. **模式可切换** — 同一框架通过模式配置适配不同创作领域
7. **创意认知必须对齐** — 每个关键决策点使用 **creative-align** 技能，确保AI的理解与用户一致，产出 `docs/creative-bible.md` 作为全团队认知基准
8. **人物必须有心理弧线标尺** — 每个有姓名角色必须有完整的人物小传，包含核心价值观排序、知情范围、心理弧线节点，作为人物行为是否跳脱的标尺。人物行为必须符合小传中的心理设定，不得为推进剧情而牺牲人物逻辑

9. **每输出必质检（默认完整版）** — 在默认全流程模式下，每个阶段的内容生产完成后，必须经过「editor 一致性审查 → script-expert 对白/逻辑审查 → quality-inspector 品质定级」的完整质检闭环。质检结果必须展示给用户，不得静默归档

10. **强制阶段过渡** — 阶段间过渡不能依赖主agent的自觉。每个阶段完成后，自动进入下一阶段的前置检查：阶段3（分章创作）完成后自动触发阶段4（全书审校 chain）；阶段4完成后自动触发阶段5（终稿整理）。主agent不得跳过或合并阶段

11. **禁止元引用** — 正文中不得使用"第X章"、"ChX"等作品内部结构标记来指涉故事事件。角色不知道自己在哪一章。所有跨事件引用必须使用角色能感知的自然叙述（如"那个翻墙的夜晚"而非"Ch15"或"第十五章"）

12. **文笔多样性** — 全书范围内避免以下重复模式：(1)同一比喻句式结构在20行内重现（如"像是一个在X了太久，终于Y"）；(2)同一肢体动作在不同章节中原样复现3次以上（如"额头抵着额头，鼻尖蹭着鼻尖"）；(3)同一修辞手法的连续堆叠。writer 交付前须自查比喻密度和动作重复度

13. **符号克制** — 正文中避免滥用破折号（——）、省略号（……）、箭头（→）等符号。(1)一段文字中破折号使用不宜超过2次，优先用逗号、冒号、句号替代；(2)省略号不宜连续使用，叙事性段落中应优先用文字描述停顿；(3)箭头（→）仅用于创作笔记和结构文档，不得出现在正文中。editor 审查时须将过度使用的符号替换为规范标点

14. **全篇质量统一** — 同一作品的所有章节必须保持一致的品质基线，不得出现前半部精工、后半部敷衍的质量断层。
    (1) subagent 批量生产时，必须向其提供已完成章节的篇幅样本（至少3章的平均行数/字数），作为品质锚点
    (2) writer 交付后，editor 须对比已完成章节的平均篇幅和质量密度，低于基线的章节退回重写
    (3) 任何一章的篇幅不得低于已完成章节平均值的60%，质量定级不得低于已完成章节的平均级别
    (4) 串行创作优于并行注水：当并行生产无法保证质量统一时，降级为串行逐章生产，每章完成后经用户确认再进入下一章

15. **质检不可中断（默认完整版）** — 从第一章到最后一章，每章必须经历完整的质检闭环（editor + script-expert + quality-inspector），不得以任何理由跳过任何一章的质检。
    (1) 每章的质检报告（editor-review、script-review、quality）是阶段过渡的必要条件，缺少任何一份即阻塞流水线
    (2) 主agent 须维护一份「质检进度表」，每完成一章的质检即在表上标注。批量生产时，质检进度落后于生产进度超过3章时，必须暂停生产、优先补质检
    (3) 全书终审（阶段4）开始前，主agent 须逐章核验质检报告完整性。缺少质检报告的章节必须先补检，不得进入终审

16. **按目标裁剪流程（短片剧本可走精简链）** — 当用户明确目标是“仅生成短片剧本（不生产视觉素材）”且要求快速出稿时，可切换到精简链路：
   (1) **极速版（3步）**：creative-director → creative-scriptwriter → creative-script-expert
   (2) **平衡版（5步）**：creative-director → creative-worldbuilder → creative-scriptwriter → creative-script-expert → creative-editor
   (3) 精简链默认跳过 creative-producer、quality-inspector 和多轮往返复审
   (4) 即使走精简链，也不得跳过 creative-director（方向把控）与 creative-script-expert（逻辑/对白质检）

## 支持的模式

| 模式ID | 名称 | 适用场景 | 核心生产工具 |
|--------|------|----------|-------------|
| `novel` | 小说创作 | 长篇/短篇小说、网文、剧本 | LLM文本生成 |
| `film` | 虚拟影视 | 短片、动画、分镜视频、概念预告片 | 生图模型 + 生视频模型 |
| `comic` | 漫画/条漫 | 漫画、条漫、绘本、分镜故事 | 生图模型 |
| `generic` | 通用创作 | 其他创作类型 | 按需配置 |

### 外部分析结果转短片剧本（film-script-only）

当用户需求是“基于外部图片分析结果快速生成短片剧本”时，使用 `film` 模式的**剧本-only**子流程，不做视觉生成。

#### 前置必做：外部数据库结果接入

本 agent 不负责图片识别。主agent 必须先接收外部系统写入的结构化结果（数据库记录），至少包含：
1. 人物信息：人数、关系、外形、情绪
2. 场景环境：地点、时间、季节、光线氛围
3. 动作情节：角色动作、互动、关键道具
4. 风格基调：治愈/悬疑/浪漫/写实等
5. 输出目标：片长、是否要对白、是否旁白主导

标准输入模板：

```plaintext
【外部图片分析结果（数据库）】
素材ID：
批次ID：
人物：
场景：
动作：
氛围：
风格参考：
需求：基于以上数据库结果生成X分钟短片分镜剧本（只要剧本，不要视觉素材）
```

#### 推荐链路

- 赶时间出初稿：`/run-chain creative.fast-script`
- 需要更稳定成稿：`/run-chain creative.balanced-script`

## 启动流程

当用户触发创作工作流时（"开始创作"、"写一部小说"、"拍一个短片"等）：

### Step 0：模式选择与项目初始化

**主agent 使用 creative-align 技能交互式执行**，与用户对齐：

1. **选择创作模式** — 根据用户意图推荐合适的模式
2. **明确创作主题** — 一句话概括作品核心
3. **确认产出目标** — 最终交付物是什么、规模多大
4. **确认质量标准** — 对质量的期望级别
5. **锚定风格方向** — 用参考作品对齐风格认知（"你希望的感觉更接近哪个作品？"）
6. **明确情感契约** — 用户希望受众获得什么核心体验

**产出**：`docs/creative-bible.md`（项目定位 + 风格定义 + 情感契约部分）

> **creative-align 在此阶段的作用：**
> - 逐个追问上述问题，每个问题给出推荐答案
> - 挑战模糊描述（"轻松搞笑具体是哪种搞笑？"）
> - 用参考作品锚定抽象风格（"赛博朋克——你更接近银翼杀手还是赛博朋克2077？"）
> - 将确认的决策即时写入 creative-bible.md

**创意总监审核**：项目概要确认后，主agent 调用 **creative-director** subagent 审核 creative-bible.md 的项目定位部分，确认：
- 创意方向是否清晰、无歧义
- 风格锚定是否足够精确（参考作品是否足够对标）
- 质量标准是否可衡量
- 项目范围是否合理（不会过大或过小）

director 产出创意评估报告，主agent 展示给用户。如有调整，先修改再进入阶段1。

**退出条件**：用户确认 creative-bible.md 中的项目定位部分，且 creative-director 审核通过后，进入阶段1。

```markdown
# 项目概要

## 创作模式
[模式名称]

## 作品主题
[一句话概括]

## 产出目标
- 最终交付物：[具体描述]
- 规模/篇幅：[章节数/时长/页数等]
- 目标受众：[谁来看/读]

## 质量标准
- 质量级别：[专业级/业余精品/实验性]
- 特殊要求：[风格偏好、禁忌、参考作品等]

## 工作目录结构
docs/          — 创意圣经、设定、大纲等文档
├── creative-bible.md  — 创意认知基准（所有决策的单一事实来源）
├── cdr/               — Creative Decision Records
├── worldbuilding.md, characters.md, story-outline.md 等
chapters/      — 章节/分场/分页内容（按模式调整）
assets/        — 视觉资产（图片、视频等）
reports/       — 质检报告、评审报告
output/        — 最终交付物
```

**退出条件**：用户确认项目概要后，进入阶段1。

## 通用阶段流程

所有模式共享相同的**阶段编排模式**，但每个阶段的具体内容、Agent角色、产出物因模式而异。

### 阶段编排规则

对于每个阶段：

1. **主agent 告知用户**：当前进入第 N 阶段（阶段名称），简要说明本阶段目标
2. **执行阶段内容**：
   - 如果是**交互式阶段**：主agent 与用户讨论、确认
   - 如果是**自动化阶段**：启动 production chain，告知用户等待
3. **展示结果**：将阶段产出展示给用户
4. **用户确认**：用户确认满意后，进入下一阶段
5. **如需修改**：根据用户反馈迭代，直到用户满意

## 阶段1.8：人物小传（所有带角色的创作模式通用）

**适用模式**：小说创作（含推理小说）、虚拟影视、漫画/条漫 — 任何涉及角色的创作类型。

**执行时机**：在谜题设计（如有）或世界观设定完成后、故事大纲之前执行。

**执行方式**：主agent 使用 **creative-align** 技能与用户逐一讨论每个角色，按模板为每个有姓名角色撰写人物小传。

### 对齐流程（creative-align）

1. **角色核心矛盾** — 逐个角色确认内心最大的冲突是什么，用场景测试（"他遇到X会怎么做？"），确保理解一致
2. **核心价值观排序** — 每个角色最看重什么（如：情义 > 名利 > 生命），排序决定角色在极端情况下的选择
3. **知情范围确认** — 每个角色知道什么、不知道什么，确保叙事的公平性和可信度
4. **心理弧线节点** — 预设角色在关键章节/场景的情绪/心理状态转折点，确保心理活动线性发展
5. **秘密层级分层** — 表层秘密（可主动展示）、中层秘密（不愿说）、深层秘密（死也不说），确保角色不突破底线
6. **对白特征确认** — 说话风格（简洁/啰嗦/隐喻/直白），用场景测试角色在不同压力下的对白变化

### 人物小传模板

每个有姓名角色必须包含以下内容：

```markdown
## [角色名] 人物小传

### 基础信息
- 身份/年龄/外貌
- 与故事的关系
- 在故事中的功能

### 核心价值观（按优先级排序）
1. 第一重要的价值
2. 第二重要的价值
3. 第三重要的价值

### 心理弧线（按章节/场景递进）
| 章节/场景 | 情绪/心理状态 | 触发事件 | 行为表现 |
|-----------|-------------|---------|---------|
| 第N章 | 初始状态 | —— | 行为A |
| 第N章 | 转变1 | 事件X | 行为B |
| 第N章 | 转变2 | 事件Y | 行为C |

### 知情范围
- 知道什么信息（按章节列出）
- 不知道什么信息（按章节列出）
- 对故事核心秘密的知情程度

### 秘密层级
- **表层秘密**（角色愿意展示的）
- **中层秘密**（角色不愿说的）
- **深层秘密**（角色死也不会说的）

### 核心矛盾
- 角色内心最大的冲突是什么？
- 这个冲突如何在故事中推进？

### 对白特征
- 说话风格（简洁/啰嗦/隐喻/直白）
- 常用措辞
- 面对压力时的说话变化

### 行为标尺
- 角色绝不会做的事：
- 角色在极端情况下可能做的事：
- 角色最害怕发生的事：
```

### 产出
- `docs/character-profiles/[角色名].md` — 每个有姓名角色一份小传
- `docs/creative-bible.md` — 更新角色核心设定部分

### 创作纪律（通用，适用于所有写作阶段）
- 分章/分场景创作时，每写一个角色的行为/对白，必须对照小传确认是否符合其心理设定
- 如果创作中发现需要调整人物设定，必须先更新小传，再更新正文
- 人物心理弧线必须线性发展，不得跳脱
- 人物行为必须有心理动因，不得为推进剧情而牺牲人物逻辑

#### 写作红线（全书强制执行）

**红线1：符号克制**
- 正文中破折号（——）不宜滥用。一段文字中破折号使用不宜超过2次，优先用逗号、冒号、句号替代
- 常见过度模式："不是X——是Y"应改为"不是X，是Y"或"不是X。是Y"；"A——B"短句应改为"A，B"或"A：B"
- 省略号（……）不宜连续使用，叙事停顿应优先用文字描述
- 箭头（→）仅用于创作笔记和结构文档，不得出现在正文中
- 对话中用于表示被打断的破折号可保留。正文中作为解释性插入语的长破折号应优先替换

**红线2：禁止元引用**
- 正文中不得出现"ChX"、"第X章"等作品内部标记来指涉情节
- 跨事件引用须用角色可感知的自然叙述（如"那晚"、"上次"、"那次在回廊"）
- 例外：章节标题、章末元数据区允许使用，但不得出现在正文中

**红线3：字符数不得幻觉**
- 正文中每出现"X个字"、"一共X个字"等字数声称时，必须逐字核验
- 核验标准：一个汉字=1个字，标点符号不计
- 字数较多（≥5字）或难以准确计数时，改用不具体计数的表达方式

**红线4：空间逻辑连贯**
- 同一场景内，角色移动路径必须完整，不得出现跳跃（如从桌边直接到床上无过渡）
- 如更换地点，须有明确的过渡描述（抱起、走向、跟随等）

**红线5：文笔多样性**
- 同一章节内，结构相同的比喻句式不得在20行以内重复出现
- 全书范围内，同一肢体动作描述不得原样出现3次以上
- 常见高风险陷阱："像是一个在X了太久的人，终于Y"、"额头抵着额头，鼻尖蹭着鼻尖"、"像一X被Y的Z"

### 退出条件
用户确认所有有姓名角色的人物小传完整且合理，进入下一阶段。

**创意总监审核**：人物小传全部完成后，主agent 调用 **creative-director** subagent 审核：
- 所有角色的心理弧线是否构成完整的戏剧张力
- 角色之间的冲突关系是否足够推动剧情
- 角色设定是否与世界观协调
- 是否每个角色都有不可替代的故事功能

director 产出角色体系评估报告，主agent 展示给用户。如需调整角色设定，先更新小传再进入下一阶段。

**退出条件**：用户确认人物小传，且 creative-director 审核通过后，进入下一阶段。

## 模式 A：小说创作

**触发条件**：用户说"写小说"、"创作故事"、"写一部xxx题材的小说"等

### 子模式：推理小说（Mystery）

**触发条件**：用户提到"推理"、"侦探"、"密室"、"谋杀案"、"阿加莎风格"、"本格推理"等

推理小说需要**特殊的前期设计流程**。核心区别：

| 维度 | 普通小说 | 推理小说 |
|------|---------|---------|
| **核心设计** | 人物弧光、情感冲突 | 谜题结构、逻辑链条 |
| **角色设计** | 动机驱动 | 动机+机会+不在场证明 |
| **大纲设计** | 起承转合 | 线索埋设→误导高潮→解答 |
| **特殊Agent** | 无 | creative-mystery-designer（推理设计师） |
| **额外产出** | 无 | mystery-core.md, clue-map.md, suspect-matrix.md |

**推理小说专属阶段**：

在**阶段2（大纲）之前**，依次插入**阶段1.5：谜题设计**和**阶段1.8：人物小传**：

```
阶段1：世界观与角色设定
    ↓
阶段1.5：谜题设计（推理小说专属）
    - creative-mystery-designer 设计核心诡计
    - 嫌疑人矩阵（动机/机会/不在场证明）
    - 线索地图（真线索vs红鲱鱼）
    - 推理路径（侦探和读者的解谜路径）
    ↓
阶段1.8：人物小传（通用-详见上方独立章节）
    - 每个有姓名角色写一份小传
    - 包含：核心价值观排序、心理弧线、知情范围、秘密层级、对白特征
    - 作为后续所有人物的行为标尺
    - 确保人物心理活动线性发展，不跳脱
    ↓
阶段2：故事大纲（融入线索埋设计划 + 人物弧线）
    ↓
阶段3：分章创作（按线索地图创作，按小传写人物）
```

**文档结构（推理小说）**：
```
docs/
├── creative-bible.md
├── mystery-core.md          # 谜题核心设计（诡计类型、解答）
├── suspect-matrix.md        # 嫌疑人矩阵
├── clue-map.md              # 线索地图（含红鲱鱼设计）
├── deduction-path.md        # 推理路径
├── character-profiles/      # 人物小传（新增目录）
│   ├── character-name.md    # 每个有姓名角色一份
│   └── ...
└── story-outline.md         # 大纲（融入线索埋设计划）
```

> **人物小传原则**：每个有姓名角色的每个行为都必须有心理动因，不能为推进剧情而牺牲人物逻辑。主agent 在分章创作阶段必须参照小传撰写人物对白和行为，确保心理弧线的线性发展。如果创作过程中发现人物需要调整，必须先更新小传，再更新正文。

### 阶段 1：世界观与角色设定（交互式）

**执行方式**：主agent 使用 **creative-align** 技能与用户对齐创意方向，调用 **creative-worldbuilder** subagent 辅助构建具体设定

**对齐流程（creative-align）：**
1. 故事背景 — 逐个追问时代、地点、社会环境，用场景测试用户的选择
2. 核心设定 — 挑战模糊的规则描述（"魔法的代价具体是什么？"）
3. 主要角色 — 用场景压力测试角色性格（"他遇到X会怎么做？"）
4. 核心冲突 — 确认冲突层次（内在/人际/社会/环境）
5. 故事基调 — 用参考作品锚定（"基调更接近哪个作品？"）

每个决策确认后即时更新 `docs/creative-bible.md`，重要决策记录为 CDR。

**产出**：
- `docs/creative-bible.md` — 更新世界观核心、角色内核部分
- `docs/worldbuilding.md` — 世界观设定（详细展开）
- `docs/characters.md` — 角色档案（详细展开）
- `docs/cdr/` — 关键创意决策记录（如有）

**后续环节**：角色设定确认后，进入**阶段1.8：人物小传**（详见上方独立章节）撰写每个有姓名角色的完整小传。

**创意总监审核**：世界观和角色设定确认后，主agent 调用 **creative-director** subagent 审核：
- 世界观设定的内在自洽性
- 角色设定是否符合故事基调
- 世界观与角色之间是否协调一致
- 整体创意的可行性和吸引力

director 产出创意评估报告，主agent 展示给用户。如有方向性问题，先调整再推进。

**退出条件**：用户确认设定文档，且 creative-director 审核通过后，进入下一阶段。

### 阶段 1.5：谜题设计（推理小说专属，交互式）

**触发条件**：创作模式为推理小说（用户确认或主agent推断）

**执行方式**：主agent 使用 **creative-align** 技能与用户对齐诡计方向，调用 **creative-mystery-designer** subagent 设计谜题结构

**对齐流程（creative-align）**：
1. **核心诡计类型** — "你想要什么类型的谜题？"（密室/不可能犯罪/叙述诡计）
2. **限制条件** — "密室的限制是什么？"（门窗反锁/众目睽睽/时间差）
3. **公平性确认** — "你希望读者有机会比侦探早猜出真相吗？"
4. **风格锚定** — "更接近阿加莎的乡村别墅，还是奎因的逻辑流？"

**设计流程（creative-mystery-designer）**：

#### Step 1：核心诡计设计
- 确定"不可能犯罪"的核心机制
- 确保物理/心理上可行
- 确保解答不"作弊"

#### Step 2：嫌疑人矩阵设计
设计 3-5 个嫌疑人，每个都有动机、机会、手段、不在场证明、红鲱鱼级别。

**设计原则**：
- **头号嫌疑人**：动机最明显，实为红鲱鱼
- **真凶**：动机隐藏深，不在场证明有巧妙漏洞
- **意外人选**：最不可能的人

#### Step 3：线索地图设计
规划真线索和红鲱鱼的埋设与揭示，确保公平性。

#### Step 4：推理路径设计
设计侦探推理链条和读者解谜路径。

#### Step 5：公平性检查
确保符合古典推理原则。

**产出**：
- `docs/mystery-core.md` — 谜题核心设计
- `docs/suspect-matrix.md` — 嫌疑人矩阵
- `docs/clue-map.md` — 线索地图
- `docs/deduction-path.md` — 推理路径
- `docs/creative-bible.md` — 更新（标注推理小说类型）

**退出条件**：用户确认谜题设计的公平性和完整性

> **重要**：推理小说的 creative-bible 必须标注为推理类型，后续所有创作必须严格遵循 clue-map.md 的线索埋设计划。

### 阶段 2：故事大纲（交互式）

**执行方式**：主agent 使用 **creative-align** 技能与用户对齐叙事方向，调用 **creative-scriptwriter** subagent 生成大纲方案

**推理小说特殊处理**（如已执行阶段1.5和1.8）：
- 大纲必须融入 `docs/clue-map.md` 的线索埋设计划
- 每个章节的核心事件必须对应线索的出现或揭示
- 大纲必须融入人物小传的心理弧线节点，确保人物发展有线性路径
- scriptwriter 必须与 creative-mystery-designer 协作，确保大纲符合推理路径
- 节奏规划要考虑"误导高潮"（第3-4章）和"真相揭示"（第7-8章）的节奏
- 分章创作时，每章的人物行为都必须对照人物小传进行一致性检查

**对齐流程（creative-align）：**
1. 叙事结构 — 确认结构类型（三幕/英雄之旅/自定义），追问为什么
2. 主线走向 — 用场景测试关键转折点（"读者在这里会有什么感受？"）
3. 结局方向 — 这是最重要的决策之一，必须精确对齐（HE/BE/OE，以及为什么）
4. 节奏规划 — 确认张力曲线走向
5. 章节划分 — 在方向确认后，由 scriptwriter 细化

**产出**：
- `docs/creative-bible.md` — 更新故事走向部分
- `docs/story-outline.md` — 详细故事大纲
- `docs/cdr/` — 关键叙事决策记录（如有）

**退出条件**：用户确认 creative-bible.md 中的故事走向，以及详细大纲

- 故事结构是否完整、有足够的戏剧张力

### 阶段 3：分章创作（流水线模式）

按 `docs/story-outline.md` 中的章节计划，**分批执行**。

#### 核心原则：流水线工艺

**写审不互等，审修不串行。**

传统串行模式（效率低）：
```
写Ch1 → 审Ch1 → 修Ch1 → 写Ch2 → 审Ch2 → 修Ch2 → ...
```

流水线模式（推荐）：
```
时间 →
写Ch6 ─→ 审Ch6 ─→ 修Ch6 ─→ 定Ch6
        写Ch7 ─→ 审Ch7 ─→ 修Ch7 ─→ 定Ch7
                 写Ch8 ─→ 审Ch8 ─→ 修Ch8 ─→ 定Ch8
```

**任何时候都有至少2-3章处于不同阶段（写/审/修/定），互不阻塞。**

---

#### 3a. 流水线启动方式（替代原 chain）

告知用户当前正在创作第 N 到第 N+M 章，然后并行启动以下任务：

**第一步：并行启动（一轮调用完成）**
```
任务A：writer → 写第N章
任务B：writer → 写第N+1章（如有）
任务C：writer → 写第N+2章（如有）
任务D：editor → 审第N-1章（与前文的衔接）
任务E：script-expert → 审第N-1章（对白/角色/逻辑）
```

> 并行数上限：writer ≤ 3（同批章节），editor+script 不限制

> ⚠️ **品质锚点机制**：批量启动 writer 前，必须向每个 writer 提供已完成章节的篇幅样本（至少最近3章的行数/字数），以及以下指令——
>    "你正在创作同一部作品中的一章，该作品已完成章节的平均篇幅约为N行/每章。你的产出不得低于该平均值的60%，内容密度和情感张力不得低于已完成章节的基准线。"
> 主agent 须在 task 中明确写入品质参照数据。editor 审查时将对比已完成章节的平均品质，低于基线的退回重写。

**第二步：审校循环（每章独立并行）**

每章的审校由 **editor + script-expert 并行执行**，互不等待：

```
editor（一致性审查）：
  ├── 设定一致性：与creative-bible、角色小传、大纲的交叉检查
  ├── 空间逻辑：人物移动路线、场景物理位置是否合理
  ├── 时间逻辑：章节内时间推移、与前后章的时间衔接
  ├── 行为逻辑：人物行为是否有前因驱动、因果链条完整
  ├── 心理逻辑：心理状态是否符合人物小传预设
  ├── 伏笔审查：伏笔是否按clue-map准确埋设/回收
  ├── 文本质感：文风一致性、民国气息（如适用）
  ├── **字符数核验**：逐条核验本章节所有"X个字"声称，
  │   发现错误立即修正。修改对话时同步扫描上下文中的字数声明。
  │   若字数较多或修改后有歧义，改用不具体计数的表达方式
  │   （如"他开口道"而非"他说了三个字"）
  └── **近距离重复检测**：同一章节20行以内出现结构相同的
      比喻句式（如"像是一个人在X了太久，终于Y"）须改写其一；
      跨章节同一动作描写出现3次以上须替换意象

script-expert（对白/逻辑/情感审查）：
  ├── 角色-对白一致性（对照人物小传的对白特征）
  ├── 角色声音差异化（遮住角色名能否判断谁在说话）
  ├── 角色言行是否基于其知情范围
  ├── 情感可信度：角色的情感反应是否符合其价值观排序
  ├── 观众体验：第一遍阅读的清晰度和情感冲击力
  ├── 媒介适配：小说对白的自然度、潜台词
  └── **字数与元引用核验**：修改对白时同步扫描上下文中的
      "X个字"声明，修正不匹配计数；检查是否有"ChX"或
      "第X章"类元引用混入正文，发现即替换为自然叙述
```

**第三步：集中修复**

收集 editor + script-expert 两份审查意见，writer **一次性修复**所有问题，不来回串行：

```
writer → 读取两份审查报告 → 一次性应用所有修复
```

问题分级处理：
| 级别 | 处理方式 |
|:----:|:---------|
| 🔴 S/A级 | 立即修复，否则阻塞流水线 |
| 🟡 B级 | 建议修复，可在本批次内集中处理 |
| 🟢 C级 | 可选，留到全书终稿阶段统一处理 |

**第四步：品质定级**

修复完成后，**creative-quality-inspector** 逐章独立定级：
- 从受众视角评估整体品质
- 检查所有 S/A/B 级问题是否关闭
- 给出品质定级（S/A/B/C级）
- 产出 quality-assessment 报告

---

#### 3b. 批次规划建议

根据章节内容特征分组，同一批次内尽量保持风格一致：

| 批次 | 章节范围 | 特征 | 亲密戏 |
|:----:|:---------|:-----|:------|
| Batch 1 | Ch1-4 | 开篇·重生·献身·心死（情感启动） | 含1-2场 |
| Batch 2 | Ch5-8 | 入门·合卺·新居·归雁（环境转移） | 含1场 |
| Batch 3 | Ch9-13 | 试探·博弈·宅斗·信息战（展开） | 含1-2场 |
| ... | ... | ... | ... |

每批次启动时，上一批次的审校应已基本完成。

---

#### 3c. 结果确认（交互式）

每批次完成后，主agent：
1. 向用户展示本批次各章的摘要和审校报告
2. 展示 quality-inspector 的品质定级
3. 如有严重问题，与用户讨论处理方式
4. 用户确认满意后，进入下一批次

#### 3d. 中途调整（用户随时可触发）

用户在确认章节时，除了"OK继续"，还可以选择：

**情况 A：修改当前章节**
- 用户提出修改意见 → 主agent 启动修改链（见下方）→ 重新展示 → 用户确认

**情况 B：回退修改已完成章节**
- 用户指出第 X 章需要修改 → 主agent 评估影响范围：
  - 仅影响第 X 章本身 → 启动修改链修改第 X 章
  - 影响后续章节的衔接 → 与用户讨论后续章节是否需要同步调整
  - 影响大纲/设定 → 先更新 creative-bible.md 和 story-outline.md，再修改章节

**情况 C：调整后续方向**
- 用户想改变后续剧情走向 → 主agent 使用 **creative-align** 重新对齐方向
- 更新 creative-bible.md 和 story-outline.md
- 已完成的章节不受影响，从下一章开始按新方向创作

**修改链**（对已完成章节的修改，必须包含质检闭环）：
```
creative-writer → 根据修改意见修改目标章节
    → creative-editor → 检查修改后的一致性（与前后章节的衔接）
      ├── 空间逻辑：修改后的位置/路线是否与前后章节矛盾
      ├── 时间逻辑：修改后的时间点是否与故事内时间线吻合
      ├── 行为逻辑：修改行为是否有前因驱动
      ├── 心理逻辑：修改后的心理状态是否符合人物小传
      └── 文字冲突：是否引入了新的矛盾
    → creative-script-expert → 检查修改后的对白/角色一致性
    → creative-writer → 修复衔接问题和对白问题
    → creative-editor → 终审
    → creative-quality-inspector → 品质定级
```
> ⚠️ 修改链同样必须执行完整的质检循环（editor + script-expert + quality-inspector），不得因修改简单而跳过任何一环。

修改完成后，主agent 向用户展示修改内容和影响评估。

**阶段间检查点**：
- 每完成一章，简要汇报进度（已完成 X/N 章）
- 每完成一幕（剧情大段落），做一次阶段性确认
- 用户随时可以说"停一下，我想调整..."触发中途调整

**产出**：`chapters/chapter-NN.md`

**退出条件**：所有章节完成

### 阶段 4：全书审校（自动化 + 交互式确认）

所有章节完成后，**必须**进行全书级别的审校。不得跳过或合并到终稿整理。

**前置检查（质检完整性核验）**：
在运行终检链之前，主agent 须逐章核验：
- 每章是否存在 3 份质检报告（editor-review + script-review + quality）
- 缺少任何一份的章节须标记为「质检缺失」，立即补检
- 补检完成前，不得进入终审阶段
- 主agent 须向用户展示质检报告完整性矩阵，标注缺失项，经用户确认后方可推进

**执行方式**：

运行终检链：
```
/run-chain creative.full-work-finalization
```

chain 自动执行以下步骤（详见 `creative-full-work-finalization.chain.json`）：
1. **creative-editor** → 全书跨章节一致性审查
   - 空间逻辑、时间逻辑、行为逻辑、心理逻辑
   - 角色一致性、伏笔回收、节奏评估、设定一致性、文字冲突
   - 产出 `reports/full-review.md`
2. **creative-script-expert** → 全书对白/角色/逻辑审查
   - 跨章节的性格漂移检查、逻辑矛盾、角色声音稳定性
   - 心理弧线完成度检查
   - 产出 `reports/full-script-review.md`
3. **creative-writer** → 根据两份审校报告修复问题
4. **creative-editor** → 修复后复审

### 阶段 6：台词提炼与跨媒体适配（通用）【新增】

**适用场景**：小说/剧本/漫画正文完成后，需要将叙事文本中的对白提炼为可直接用于其他媒介（剧本、漫画分镜、影视配音等）的台词文本。

**执行方式**：主agent 逐章提取人物对白，对照人物小传中的对白特征，去除叙事性修饰，保留纯台词，按媒介需求进行格式适配。

#### Step 1：台词提取

从正文中提取所有人物对白，剥离叙述性引导（"某某说""某某道"等），只保留纯台词文本。

**转换示例**：

| 原文（小说） | 提取后（台词） |
|-------------|---------------|
| "狄大人远道而来，贫僧有失远迎。"慧明合十行礼。 | 慧明：狄大人远道而来，贫僧有失远迎。 |
| "法师便是慧明法师？"狄仁杰还礼。 | 狄仁杰：法师便是慧明法师？ |

#### Step 2：对白特征校准

对照人物小传中的「对白特征」检查每句台词是否匹配角色设定，校准不符合人设的对白：

| 角色 | 对白特征 | 校准要点 |
|------|---------|---------|
| 慧明 | 平和从容·带佛家顿挫·双关意味 | 是否过于急切？是否过于口语化？ |
| 静虚 | 直接锐利·激动时提高声调·悲伤时沉默 | 是否过于温和？是否符合"刚直"设定？ |
| 明心 | 轻声·简短·被追问会停顿 | 是否说了太多话？是否符合"内敛"设定？ |
| 郑元礼 | 热情周到·紧张时打官腔·被逼死角后沉默 | 是否过早露怯？是否符合"官员体面"设定？ |

#### Step 3：媒介格式适配

根据目标媒介输出不同格式的台词文本：

**格式一：剧本格式**（适用于影视/舞台）
```
[场景] 大慈恩寺·禅房·日
[人物] 慧明·狄仁杰

慧明：狄大人远道而来，贫僧有失远迎。
狄仁杰：法师便是慧明法师？
慧明：正是贫僧。大人请。
```

**格式二：漫画分镜格式**（适用于漫画/条漫）
```
【格1】慧明合十（全景）
对白：慧明「狄大人远道而来，贫僧有失远迎。」

【格2】狄仁杰还礼（中景）
对白：狄仁杰「法师便是慧明法师？」
```

**格式三：纯对白格式**（适用于配音/广播剧）
```
慧明：狄大人远道而来，贫僧有失远迎。
狄仁杰：法师便是慧明法师？
慧明：正是贫僧。大人请。
```

#### Step 4：台词本输出

按章节整理为完整的台词本，标注每句台词的出处章节、说话人、情绪状态：

```markdown
# 台词本：第X章

## 场景一：[场景名称]

| 说话人 | 台词 | 情绪 | 出处 |
|--------|------|------|------|
| 慧明 | 狄大人远道而来，贫僧有失远迎。 | 平和·恭敬 | 第1章 |
| 狄仁杰 | 法师便是慧明法师？ | 礼貌·审视 | 第1章 |
| 慧明 | 正是贫僧。大人请。 | 从容 | 第1章 |
```

**产出**：
- `output/dialogue/dialogue-full.md` — 完整台词本
- `output/dialogue/dialogue-[场景名].md` — 按场景拆分
- `output/dialogue/character-[角色名]-lines.md` — 按角色拆分的个人台词集

**创作纪律**：
- 台词提炼不改变原文情节和人物性格
- 若发现台词在脱离叙述后含义不清（如"那件事"之类的前指代），需补充简短的舞台指示/动作描写但不添加原文没有的信息
- 同时输出一份「台词问题清单」——对白是否符合小传？是否存在所有角色说话方式一样的问题？

**退出条件**：用户确认台词本完整、角色声音可区分、格式符合目标媒介需求

## 自动化 Production Chain 说明

所有模式共用同一个 chain：`creative.stage-production`

chain 的行为由**当前模式**决定，具体差异：

| 维度 | novel | film | comic |
|------|-------|------|-------|
| 创作者角色 | creative-writer | creative-producer | creative-producer |
| 一致性审查者 | creative-editor | creative-editor | creative-editor |
| 对白/逻辑审查者 | creative-script-expert | creative-script-expert | creative-script-expert |
| 阶段品质定级 | creative-quality-inspector | creative-quality-inspector | creative-quality-inspector |
| 输入文档 | 大纲 + 设定 + 前文 | 分镜 + 视觉设定 | 分镜脚本 + 设定 |
| 产出类型 | 文本（.md） | 图片/视频 | 图片 |
| 审查重点 | 叙事一致性 + 对白 + 品质 | 视觉一致性 + 对白 + 品质 | 画面一致性 + 对白 + 品质 |
| 生产工具 | LLM | 生图/生视频模型 | 生图模型 |

**chain 结构**（所有模式统一，9步完整质检闭环）：
```
创作/生产 → editor一审 → script-expert对白/逻辑审查 → 整合修改 → editor二审 → 最终修改 → editor终审 → quality-inspector阶段品质定级
```

> ⚠️ **强制规则**：此 chain 的每个步骤不可跳过、不可省略。主agent 调用 `/run-chain creative.stage-production` 时自动执行全部9步。如有特殊原因需要跳过某一步，必须先向用户说明原因并获得用户确认。

**全书终检链**（阶段4强制使用）：
```
/run-chain creative.full-work-finalization
```
结构：
```
editor全书一致性审校 → script-expert全书对白/逻辑审查 → writer修复 → editor复审 → quality-inspector最终品质定级
```

## 判断标准速查表

### 必须交互式（主agent执行）

| 场景 | 原因 |
|------|------|
| 模式选择与项目概要 | 需要用户决定创作方向 |
| 世界观/设定/视觉风格 | 需要用户确认创意方向 |
| 大纲/分镜脚本 | 需要用户确认结构 |

| 全书/全片审查结果 | 用户决定是否接受 |
| 最终交付确认 | 用户确认成品 |

### 可以自动化（chain/subagent执行）

| 场景 | 原因 |
|------|------|
| 基于已确认设定的章节创作 | 输入明确，可自动执行（结果需确认） |
| 基于已确认分镜的镜头生产 | 输入明确，可自动执行（结果需确认） |
| 一致性/连续性审查（editor） | 基于设定文档的标准评估 |
| 对白/逻辑审查（script-expert） | 基于人物小传的客观标准 |

| 审查-修改循环 | 自动化闭环，直到通过 |
| 阶段品质定级（quality-inspector） | 基于多份审查报告的综合评估 |

### ⛔ 禁止跳过（强制执行的环节）

| 环节 | 强制理由 |
|------|---------|

| **full-work-finalization** 全书终检链（阶段4） | 跨章节全景审查，不可合并到终稿整理 |

## 工作流 B：反馈驱动迭代

**触发条件**：用户说"前N章发布后收到读者反馈"、"根据反馈调整后续"、"读者说角色B太讨厌了，需要改"等

这和 dev-team 的"功能迭代"是同一个模式——在已发布作品的基础上，根据外部反馈进行系统性调整。

### 与创作过程中"中途调整"的区别

| 场景 | 阶段3c 中途调整 | 工作流B 反馈迭代 |
|------|---------------|----------------|
| 触发时机 | 创作过程中 | 部分章节已发布 |
| 反馈来源 | 用户自己的判断 | 读者/观众的外部反馈 |
| 影响范围 | 单章或后续方向 | 可能回改已发布章节 + 调整后续 |
| 文档更新 | 可选更新 creative-bible | 必须更新 creative-bible 记录反馈 |

### 阶段 1：反馈整理与影响评估（交互式）

**执行方式**：主agent 使用 **creative-align** 技能，逐个追问每条反馈

**对齐流程：**
1. **反馈收集** — 用户提供读者反馈（可以是一条或多条）
2. **逐个追问** — 对每条反馈：
   - "这条反馈具体是指哪个章节/角色/情节？"
   - "你认同这条反馈吗？为什么？"
   - "如果不改，对作品有什么影响？"
   - "如果改，需要改什么层面？"
3. **影响分层判断** — 对每条认可的反馈，判断影响层级：
   - **L1 表层** — 单章文字/画面调整，不影响整体
   - **L2 角色层** — 需要调整角色设定或表现
   - **L3 情节层** — 需要调整大纲或后续走向
   - **L4 核心层** — 影响 creative-bible 的核心决策

**产出**：
- `docs/reader-feedback.md` — 结构化反馈记录
- `docs/feedback-impact.md` — 影响评估报告

```markdown
# 读者反馈影响评估

## 反馈清单

### 反馈 1：读者说"角色B出场太突兀"
- **来源**：第3章评论区
- **用户态度**：✅ 认同 / ❌ 不认同
- **影响层级**：L2 角色层
- **处理方案**：
  - 修改章节：第3章（增加铺垫）
  - 影响范围：第3章 + 第4-5章（调整角色B的提及方式）
  - 设定调整：角色B的背景需要在第2章埋设伏笔
  - creative-bible 更新：角色B的"初次亮相"决策更新

### 反馈 2：读者说"节奏太慢，想弃书"
- **来源**：第5章后读者流失
- **用户态度**：✅ 认同
- **影响层级**：L3 情节层
- **处理方案**：
  - 修改章节：第5章（加快节奏）
  - 影响范围：第5-8章（调整情节密度）
  - 大纲调整：将原计划第8章的高潮提前到第6章
  - creative-bible 更新：节奏规划部分更新

## 综合影响评估
- **最大影响层级**：L3（情节层）
- **需要回改的已发布章节**：[第3章、第5章]
- **需要调整的后续章节**：[第4-8章]
- **需要更新的文档**：[creative-bible.md, story-outline.md]
- **建议方案**：先更新文档，再执行修改
```

**退出条件**：用户确认影响评估，明确要处理哪些反馈

---

### 阶段 2：创意圣经更新（交互式）

**执行方式**：主agent 使用 **creative-align** 技能

**对齐流程：**
1. 逐条讨论需要回写 creative-bible 的反馈
2. 确认每个核心决策是否需要调整
3. 更新 creative-bible.md，标注变更原因（"根据读者反馈，角色B的..."）
4. 如有必要，记录新的 CDR

**产出**：
- 更新后的 `docs/creative-bible.md`（带变更标注）
- 新的 `docs/cdr/` 文件（如有新的方向性决策）

**退出条件**：用户确认更新后的 creative-bible

---

### 阶段 3：文档级联更新（自动化/交互式）

根据 creative-bible 的变更，自动或人工更新下游文档：

| 影响层级 | 需要更新的文档 | 更新方式 |
|---------|--------------|---------|
| L1 表层 | 目标章节本身 | 直接修改 |
| L2 角色层 | characters.md + 涉及该角色的所有章节 | subagent 辅助 |
| L3 情节层 | story-outline.md + 涉及章节 | subagent 辅助 |
| L4 核心层 | worldbuilding.md / visual-design.md 等 | subagent 辅助 |

**产出**：
- 更新后的 `docs/story-outline.md`（如有情节调整）
- 更新后的 `docs/characters.md`（如有角色调整）
- 更新后的 `docs/worldbuilding.md` / `docs/visual-design.md`（如有核心设定调整）

**退出条件**：所有下游文档更新完成

---

### 阶段 4：已发布章节回改（循环执行）

对需要回改的已发布章节，逐章执行修改链。

#### 4a. 评估修改风险

主agent 评估：
- 修改后与前后章节的衔接是否自然
- 是否有伏笔/线索需要同步调整
- 读者已读过的内容，修改后会不会造成困惑

#### 4b. 执行修改链

```
creative-writer → 根据反馈修改目标章节
    → creative-editor → 检查修改后的一致性（与前后章节的衔接）
    → creative-writer → 修复衔接问题
    → creative-editor → 检查与读者反馈的对齐度
    → creative-editor → 终审
```

**注意**：
- 已发布章节的修改要谨慎，评估是否值得（有时"将错就错"后续圆回来比强行修改更好）
- 修改后要更新章节元数据（`chapters/chapter-NN.md` 中的修改记录）

**产出**：`chapters/chapter-NN-revised.md`（或覆盖原文件，保留备份）

**退出条件**：所有需要回改的章节处理完成

---

### 阶段 5：后续章节调整（循环执行）

按更新后的大纲和 creative-bible，继续创作后续章节。

**注意**：
- 后续章节必须基于更新后的设定和方向
- 如果已有部分后续章节草稿，需要检查是否需要调整
- 每章完成后同样需要用户确认

**退出条件**：全书完成

---

### 阶段 6：差异说明（交互式）

如有已发布章节的修改，向用户生成差异说明：
- 哪些章节修改了
- 修改了什么内容
- 对读者阅读体验的影响
- 建议如何告知读者（如："第3章微调了角色B的出场方式，已读读者可重读或继续"）

**产出**：`docs/revision-notes.md`

## 注意事项

1. **不要跳过确认阶段** — 即使用户说"随便你"，也要给出具体方案让用户确认
2. **视觉内容要展示** — 生图/生视频的产出必须展示给用户看，不能只输出文件路径
3. **文本内容要朗读关键段落** — 小说章节要展示精彩片段给用户预览
4. **chain 运行期间不要干预** — 让单阶段自动化流程跑完
5. **审查报告要展示给用户** — 不要自动决定"质量够了"
6. **进度要持续汇报** — 每个阶段完成后告知用户整体进度
7. **模式间可以混合** — 如小说配插图（novel + 生图），在通用模式中自定义
8. **资产文件要有组织** — 图片/视频按场景/章节分类存放，便于管理
9. **保留迭代历史** — 不要覆盖前一版本，便于回溯和对比
